## 一、文档概述

### 1.1 背景
工具型情侣应用最大的敌人不是竞品，而是「新鲜感消退」。当用户完成餐厅记录、纪念日设置、心愿单添加后，产品容易沦为「沉默的数据库」。真正能够长期留住用户的，不是功能本身，而是产品持续帮助用户「感受到被爱」和「表达爱意」的能力。

### 1.2 核心理念
> **产品不应只是记录爱情的容器，而应是爱情的催化剂。**

### 1.3 设计原则
1. **微小但具体**：一点点用心的小细节，胜过宏大的概念
2. **双向感知**：让双方都能感受到「TA 在意我」
3. **仪式常态化**：将特殊日子的仪式感渗透到日常
4. **回忆可触碰**：让过去的美好随时能被唤醒
5. **脆弱被接纳**：在关系低谷期，产品也能提供恰到好处的支持

```
优先级: P0
预估工时: 8 人日（后端 4，推送服务 2，前端 2）
```

#### 页面原型

```
早安 Push 卡片
┌─────────────────────────────────────┐
│ 💌 早安，新的一天开始了             │
│                                     │
│ "距离你们相恋 200 天还有 3 天，     │

│                                     │
│ [回复早安] [查看今日任务]           │
└─────────────────────────────────────┘

App 内早安互动页
┌─────────────────────────────────────┐
│ 早安 ☀️                             │
│                                     │
│ 你已经连续和 TA 互道早安 7 天       │
│ [早睡早起] 勋章已解锁 🏅            │
│                                     │
│ [发送早安]                          │
│ [发送早安语音]                      │
│                                     │
│ 历史早安语音库：                     │
│ ▶️ "早安宝贝，今天也要开心"        │
│ ▶️ "起床啦，太阳晒屁股了"          │
└─────────────────────────────────────┘
```

**交互流程**：
1. 用户在「我的」-「早安晚安设置」中设定叫醒时间（默认 07:30）
2. 到点后，Push 中心向双方发送早安卡片 Push
3. 点击 Push 进入 App，弹出快捷回复框
4. 双方均发送早安后，当日「甜蜜成就」解锁
5. 连续 7 天解锁「早睡早起」勋章

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★★ | 定时推送技术非常成熟 |
| 实现复杂度 | 低 | 本质是定时任务 + 推送 + 状态记录 |
| 外部依赖 | 中等 | 依赖推送服务商（个推/极光/微信模板消息） |
| 数据安全 | 低 | 无敏感信息 |
| 性能风险 | 低 | 推送量可控，无需复杂计算 |

**技术方案**：
- **定时任务**：Spring Boot Scheduler / Quartz，每日按用户设定时间扫描并触发推送
- **Push 文案动态生成**：基于当天日期、天气、纪念日倒计时组装个性化文案
- **状态记录**：`couple_daily_greeting` 表记录每日早安/晚安发送状态（who_sent, send_time）
- **语音存储**：早安语音存 OSS，元数据 `voice_greeting` 表关联 couple_id 和日期
- **语音播放**：H5 用 HTML5 audio，小程序用 innerAudioContext

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| greeting_setting_save | 保存叫醒时间 | greeting_type, set_time | 设置率 |
| greeting_push_arrive | Push 到达 | greeting_type | Push 到达率 |
| greeting_send_text | 发送文字问候 | greeting_type | 文字发送率 |

```
└─────────────────────────────────────┘
```

**甜蜜炸弹生成算法**：
1. 每日凌晨扫描所有 couple，从事件池中选择 1-2 条适合今日推送的内容
2. 事件优先级：纪念日 > 历史约会 > 投喂记录 > 笔记摘录 > 通用祝福
3. 去重控制：同一类型炸弹 7 天内不重复推送
4. 推送时间优化：选择双方历史活跃时间的交集时段（如：12:00 或 20:00）

**技术方案**：
- 内容池：`sweet_bomb_pool` 预先生成各 couple 的候选内容
- 定时任务：每日 10:00/18:00 两次扫描，触发符合条件的推送
- 文案模板：
  ```
  回忆炸弹：「{years}年前的今天，你们在{restaurant}吃了{food}，还记得吗？」
  数据炸弹：「过去 {days} 天，TA 给你投喂了 {count} 次，被接住了 {accepted} 次」
  ```
- 生成方式：规则模板为主，特殊场景才调用 LLM

**低迷检测算法**：
```python
def detect_relationship_weather(couple_id):
    last_3d = get_interaction_count(couple_id, days=3)
    last_7d = get_interaction_count(couple_id, days=7)
    last_14d = get_interaction_count(couple_id, days=14)

    if last_3d == 0 and baseline_7d_avg > 2:
        return "微风"  # 轻度
    if last_7d == 0 and baseline_30d_avg > 5:
        return "阴天"  # 中度
    if last_14d <= 1 and baseline_30d_avg > 8:
        return "小雨"  # 重度
    return "晴朗"
```

**技术方案**：
- 每日凌晨执行扫描任务，评估所有 couple 的互动状态
- 干预策略：仅向最近 3 天内有登录的一方推送（避免骚扰完全离线方）
- 推送冷却期：同一 couple 同一等级预警 7 天内只推送一次
- 数据模型：`relationship_weather_log` 表记录每次预警的时间、等级、干预方式、结果

| weather_alert_push_click | 点击预警 Push | alert_level | Push 点击率 |
| weather_intervention_copy | 复制破冰文案 | alert_level | 文案使用率 |
