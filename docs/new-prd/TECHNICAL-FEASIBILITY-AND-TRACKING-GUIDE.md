## 一、技术可行性评估框架

### 1.1 评估维度

对每个需求从以下 5 个维度进行技术评估：

| 维度 | 评分标准 | 说明 |
|------|----------|------|
| 技术成熟度 | 1-5 星 | 市面上是否有成熟方案 |
| 实现复杂度 | 低/中/高 | 开发工作量和技术栈要求 |
| 外部依赖 | 少/中/多 | 是否需要第三方 API/服务 |
| 数据安全 | 低/中/高 | 隐私风险和合规要求 |
| 性能风险 | 低/中/高 | 对现有系统性能的影响 |

### 1.2 关键技术栈预研

#### AI 能力层

| 能力 | 可选方案 | 推荐方案 | 成本估算 |
|------|----------|----------|----------|
| 文本生成 | 通义千问 / Claude / OpenAI | Claude 3.5 Sonnet (API) | ~$0.003/1K tokens |
| 图像识别 (OCR) | 阿里云 OCR / 百度 OCR / Tesseract | 阿里云 OCR | ~¥0.01/次 |
| 菜品/场景识别 | 阿里云图像识别 / 百度 AI | 阿里云图像识别 | ~¥0.02/次 |
| 语音识别 | 阿里云 ASR / 科大讯飞 | 阿里云 ASR | ~¥0.006/秒 |
| 视频生成 | 剪映 API / 即梦 / Pika | 剪映 API (优先) | 按需询价 |
| 表情包/Q版生成 | Stable Diffusion / Midjourney API | Stable Diffusion (自部署) | 服务器成本 |
| 推荐引擎 | 自研规则引擎 + 协同过滤 | 自研规则引擎 MVP → 引入阿里云 PAI | 渐进投入 |

#### 实时通信层

| 能力 | 可选方案 | 推荐方案 | 说明 |
|------|----------|----------|------|
| WebSocket | Netty + Redis Pub/Sub | Netty + Redis | 与现有 Spring Boot 栈一致 |
| 推送服务 | 微信模板消息 / 个推 / 阿里云推送 | 个推 + 微信服务号 | 覆盖 iOS/Android/H5 |
| 地图服务 | 高德地图 JS API / 腾讯地图 | 高德地图 | H5 + 小程序统一 |

#### 数据存储扩展

| 数据类型 | 推荐存储 | 原因 |
|----------|----------|------|
| 结构化数据 | MySQL + TiDB (远期) | 关系型数据为主 |
| 缓存/会话 | Redis Cluster | 已使用，扩展即可 |
| 消息/聊天记录 | MongoDB | 文档型，灵活扩展 |
| 图片/视频 | 阿里云 OSS / 七牛云 | CDN 加速 + 低成本 |
| 搜索/日志 | Elasticsearch | 日志分析、内容搜索 |
| 时序数据 | InfluxDB / Prometheus | 埋点数据、温度指数趋势 |

## 二、通用数据埋点规范

### 2.1 埋点框架选型

**推荐方案**: 混合埋点（自动埋点 + 手动关键事件埋点）

- **前端**: 封装埋点 SDK，基于 Vue 路由自动采集页面 PV/UV，关键按钮/卡片手动埋点
- **后端**: 在 Controller 层通过 AOP 拦截关键 API，记录业务事件
- **数据流转**: App/Sdk → 埋点网关 → Kafka → Flink 清洗 → ClickHouse/ES → BI 报表

**MVP 简化方案**（资源有限时）:
- 前端手动埋点 + 后端 API 日志 → 定时任务清洗 → MySQL 报表表
- 或使用神策/友盟 SaaS 服务快速启动

### 2.2 事件命名规范

```
{页面/模块}_{动作}_{对象}_{结果}

示例:
- home_click_recommend_card_success
- feed_send_gift_button_click
- tree_water_daily_task_complete
- quiz_answer_question_submit
```

### 2.3 通用事件属性

每个事件必须携带：

```json
{
  "event_id": "home_click_recommend_card",
  "timestamp": 1711843200000,
  "user_id": "u_123456",
  "couple_id": "c_789012",
  "device_id": "d_abcdef",
  "platform": "h5|mp-weixin|ios|android",
  "app_version": "1.2.0",
  "network": "wifi|4g|5g",
  "city": "上海",
  "page_path": "/pages/home/index"
}
```

### 2.4 核心指标体系

#### 增长指标 (Growth)

| 指标名 | 计算方式 | 采集事件 |
|--------|----------|----------|
| DAU | 当日独立活跃用户 | app_launch |
| MAU | 当月独立活跃用户 | app_launch |
| 新增用户 | 当日首次注册用户数 | user_register |
| 绑定转化率 | 注册且绑定情侣 / 注册总数 | couple_bind |
| 邀请转化率 | 通过邀请码注册且绑定 / 邀请链接点击 | invite_register |
| 分享次数 | 各类分享按钮点击总数 | share_click |
| 分享带新 | 通过分享链接/海报注册的用户数 | share_register |

#### 活跃指标 (Engagement)

| 指标名 | 计算方式 | 采集事件 |
|--------|----------|----------|
| 日均启动次数 | app_launch 总数 / DAU | app_launch |
| 平均使用时长 | 所有会话时长总和 / 会话数 | session_start / session_end |
| 功能使用率 | 使用某功能的 UV / DAU | 各功能入口点击 |
| 任务完成率 | 完成任务数 / 任务曝光数 | task_expose / task_complete |
| 爱心树互动率 | 浇水/施肥用户 / DAU | tree_interact |
| 投喂参与率 | 当日发送/接收投喂用户 / DAU | feed_send / feed_receive |

#### 留存指标 (Retention)

| 指标名 | 计算方式 | 采集事件 |
|--------|----------|----------|
| 次日留存 | 次日仍启动的当日新增 / 当日新增 | app_launch |
| 7 日留存 | 7 日后仍启动的 / 当日新增 | app_launch |
| 30 日留存 | 30 日后仍启动的 / 当日新增 | app_launch |
| 情侣关系留存 | 绑定 30 天后仍未解绑 / 新增绑定 | couple_unbind |

#### 商业化指标 (Monetization)

| 指标名 | 计算方式 | 采集事件 |
|--------|----------|----------|
| ARPU | 总收入 / DAU | pay_complete |
| 付费转化率 | 付费用户 / DAU | pay_complete |
| 会员续费率 | 续费订单 / 到期订单 | member_renew |
| CPS 佣金收入 | 第三方平台回传 | cps_complete |

### 2.5 埋点评审 Checklist

每个 PRD 功能在开发前必须确认：

- [ ] 功能入口是否埋点（曝光 + 点击）
- [ ] 核心流程是否埋点（步骤漏斗）
- [ ] 成功/失败状态是否区分
- [ ] 是否携带业务属性（如：推荐位 ID、任务类型、海报模板）
- [ ] 是否涉及 A/B 测试分流标识
- [ ] 异常路径是否补充埋点

## 三、通用技术风险与应对

### 3.1 AI 类功能共性风险

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| API 响应慢 | 用户体验差 | 异步预生成 + 缓存 + 降级文案库 |
| API 成本高 | 规模化后费用激增 | 自建轻量模型替代高频调用；分层调用策略 |
| 生成内容不当 | 合规风险 | 敏感词过滤 + 人工审核抽样 + 用户举报机制 |
| 数据隐私争议 | 用户投诉/下架 | 隐私授权二次确认 + 本地化处理 + 独立开关 |

### 3.2 实时通信类共性风险

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| WebSocket 连接不稳定 | 消息丢失/延迟 | 心跳检测 + 断线重连 + 离线消息补发 |
| 消息并发高峰 | 服务器压力大 | 消息队列削峰 + 连接数限流 |
| 消息安全 | 隐私泄露 | 端到端加密（E2EE）+ 消息阅后即焚选项 |

### 3.3 地图/多媒体类共性风险

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 地图 SDK 兼容性问题 | 部分机型白屏 | 提供降级静态图方案 |
| 视频/图片生成耗时 | 用户等待焦虑 | 异步生成 + 进度条 + 完成推送 |
| 存储成本激增 | OSS 费用超预算 | 图片压缩/WebP转换 + 生命周期管理 |

## 四、基础设施建议

### 4.1 新增中间件/服务清单

| 服务 | 用途 | 优先级 | 备注 |
|------|------|--------|------|
| MongoDB | 消息、胶囊、心动时刻等非结构化数据 | P1 | 可先用 MySQL JSON 字段过渡 |
| Elasticsearch | 内容搜索、日志分析 | P1 | 中小规模可用 MySQL like 过渡 |
| Kafka | 埋点数据、事件总线 | P1 | 中小规模可用 Redis List 过渡 |
| Flink | 实时数据清洗 | P2 | 日均 < 100w 事件可离线处理 |
| MinIO / OSS | 对象存储 | P0 | 直接接入阿里云 OSS |
| 消息推送服务 | Push、模板消息 | P0 | 个推/极光 + 微信服务号 |

### 4.2 团队配置建议

按当前 5 个方向同时推进，建议的团队配置：

| 角色 | 人数 | 负责方向 |
|------|------|----------|
| 后端工程师 (Java) | 3-4 人 | 业务 API、推荐引擎、消息系统 |
| 前端工程师 (Vue/UniApp) | 2-3 人 | H5、小程序、动画交互 |
| AI/算法工程师 | 1-2 人 | 推荐模型、NLP、图像处理对接 |
| 测试工程师 | 1-2 人 | 功能测试、性能测试、自动化 |
| 产品经理 | 1-2 人 | 需求细化、项目管理、数据分析 |
| UI/UX 设计师 | 1-2 人 | 视觉设计、动效设计、海报模板 |
| 运营/增长 | 1-2 人 | 活动策划、KOC、社群运营 |

**总计**: 10-15 人团队，支撑约 6 个月的第一阶段开发。
