## 一、文档概述

### 1.1 背景
情侣类工具型应用普遍存在"记录完就遗忘"的问题，用户活跃度随时间推移快速衰减。参考 Duolingo、蚂蚁森林、Keep 等产品的成功经验，将核心行为游戏化，能够显著提升用户打开频次和互动深度。

### 1.2 目标
- 将 DAU/MAU 比率从当前的工具型均值（约 15%）提升至 35%
- 提高情侣双方的同步活跃率（双方都活跃的占比）
- 通过虚拟成就系统增强用户粘性，为后续付费转化铺路

### 1.3 设计原则
1. **正向竞争**：没有情侣排行榜，避免比较压力
2. **共同目标**：所有任务和挑战都是"我们"一起完成
3. **即时反馈**：每次互动都有可见的进度增长和视觉奖励
4. **轻游戏化**：不增加使用负担，游戏元素融入日常行为

## 二、情侣等级与成长体系

### 2.1 爱心树/爱情树系统

```
需求ID: GAM-001
需求名称: 情侣爱心树
优先级: P0
预估工时: 12 人日（后端 6，前端 4，动画/视觉 2）
```

#### 核心概念
每对情侣绑定后共同获得一棵「爱心树」。日常互动行为会为爱心树积累养分，爱心树随着等级提升不断成长、开花、结果，解锁新的装扮和特效。

#### 爱心树成长机制

| 等级 | 名称 | 形态 | 解锁特权 | 升级所需养分 |
|------|------|------|----------|-------------|
| Lv.1 | 萌芽 | 刚破土的小芽 | 基础功能 | 0 |
| Lv.5 | 幼苗 | 长出嫩叶 | 解锁第一个树洞皮肤 | 500 |
| Lv.10 | 小树 | 枝丫舒展 | 解锁双人专属头像框 | 1500 |
| Lv.20 | 繁树 | 枝繁叶茂 | 解锁花园场景 | 4000 |
| Lv.50 | 花树 | 首次开花 | 解锁纪念日特效 | 15000 |
| Lv.99 | 神树 | 流光溢彩 | 解锁传说级皮肤 + 实体礼物资格 | 50000 |

#### 养分获取规则

| 行为 | 获得养分 | 每日上限 |
|------|----------|----------|
| 每日首次打开 App（双方） | +10 | 1 次 |
| 发送投喂 | +15 | 3 次 |
| 接收投喂并确认 | +10 | 3 次 |
| 新增一家餐厅到菜单 | +20 | 2 次 |
| 发布美食笔记 | +25 | 2 次 |
| 完成一个心愿 | +50 | 无上限 |
| 消息中心互发消息 | +5 | 20 条 |
| 连续签到 7 天额外奖励 | +100 | 每周 1 次 |
| 纪念日当天互动 | +200 | 无上限 |

#### 页面原型

```
首页 - 爱心树展示区（占屏幕 40%）
┌─────────────────────────────────────┐
│                                     │
│           🌳 [爱心树动画]            │
│            Lv.12 小树               │
│     当前养分：892 / 1500            │
│     [==========>        ]           │
│                                     │
│     [浇水] [施肥] [装饰] [树洞]     │
│                                     │
│   🍃 下拉摇树，看看有什么惊喜~       │
│                                     │
├─────────────────────────────────────┤
│ 今日任务  |  最近动态               │
└─────────────────────────────────────┘
```

**交互流程**：
1. 首页加载时，通过 Lottie/Canvas 渲染当前等级的爱心树动画
2. 用户完成任意可获养分行为后，播放「+XX 养分」飘字动画
3. 养分达到升级阈值时，播放升级全屏动画并解锁新装扮
4. 点击「浇水」消耗 1 次浇水机会，获得 30min 成长加速 buff
5. 下拉首页触发「摇树」彩蛋：随机掉落一条历史回忆 tips

#### 技术可行性分析

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★★ | 成长数值系统非常成熟，动画可用 Lottie |
| 实现复杂度 | 中等 | 需要设计等级公式、动画资源管理、养分事件监听 |
| 外部依赖 | 少 | 无强外部依赖 |
| 数据安全 | 低 | 游戏数值无敏感信息 |
| 性能风险 | 低 | Lottie 动画经优化后移动端可流畅运行 |

**技术方案**：
- **动画渲染**：Lottie JSON 动画，每个等级对应一组动画文件；低端机降级为静态 SVG
- **养分系统**：基于事件驱动，各业务模块（投喂、笔记、消息）完成后发送 Domain Event，爱心树服务消费并累加养分
- **等级计算**：采用渐进式升级曲线，公式 `required_nutrient = 100 × level^1.5`
- **缓存**：Redis 缓存当前 couple 的等级、养分、buff 状态
- **数据库表**：`couple_tree`（等级、总养分）、`tree_nutrient_log`（养分变动日志）

|--------|----------|----------|----------|
| tree_home_expose | 首页爱心树出现在视口 | tree_level, current_nutrient | 爱心树曝光 |
| tree_water_click | 点击浇水 | has_buff, buff_multiplier | 浇水次数 |
| tree_fertilize_click | 点击施肥 | fertilizer_type | 施肥次数 |

| tree_level_up | 爱心树升级 | old_level, new_level | 升级事件 |
| tree_nutrient_gain | 获得养分 | source_action, nutrient_amount | 养分来源分布 |
| tree_decorate_click | 点击装饰 | decoration_id | 装饰使用率 |

### 2.2 情侣段位系统

```
需求ID: GAM-002
需求名称: 情侣段位系统
优先级: P1
预估工时: 8 人日（后端 5，前端 2，规则设计 1）
```

#### 段位设计
根据「恋爱温度指数」和「近 30 天互动频次」综合评定段位：

| 段位 | 名称 | 晋级条件 | 保级条件 |
|------|------|----------|----------|
| 青铜 | 初识 | 绑定情侣后默认 | - |
| 白银 | 相知 | 连续 7 天均有互动 | - |
| 黄金 | 热恋 | 连续 30 天互动 + 温度 > 60 | 连续 14 天无互动触发保级 |
| 铂金 | 默契 | 连续 60 天互动 + 温度 > 70 | 连续 14 天无互动触发保级 |
| 钻石 | 伴侣 | 连续 90 天互动 + 温度 > 80 | 连续 14 天无互动触发保级 |
| 星耀 | 灵魂伴侣 | 连续 180 天互动 + 温度 > 85 | 连续 14 天无互动触发保级 |
| 王者 | 神仙眷侣 | 连续 365 天互动 + 温度 > 90 | 连续 14 天无互动触发保级 |

#### 保级挑战流程

```
系统检测到连续 14 天无有效互动
         ↓
向活跃方推送保级挑战通知
"你们的段位即将从钻石降到铂金，
 3 天内完成 3 次投喂即可保级！"
         ↓
用户完成 3 次投喂
         ↓
保级成功，段位维持不变
         ↓
挑战失败，自动降级并发送温情提醒
```

**技术方案**：
- 每日凌晨定时任务扫描所有 couple 的近 14 天互动记录
- 满足降级条件的发送保级挑战消息（写入消息表 + Push）
- 保级挑战状态写入 Redis，TTL=72h
- 降级/晋级时同步发送双方 Push 通知

#### 数据埋点方案

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| rank_promote | 段位晋升 | old_rank, new_rank | 晋升率 |
| rank_demote | 段位降级 | old_rank, new_rank, reason | 降级率 |
| rank_challenge_start | 保级挑战开始 | current_rank | 挑战触发率 |
| rank_challenge_success | 保级成功 | current_rank | 挑战成功率 |
| rank_challenge_fail | 保级失败 | current_rank | 挑战失败率 |
| rank_privilege_use | 使用段位特权 | privilege_type | 特权使用率 |

## 三、任务与挑战系统

### 3.1 每日情侣任务

```
需求ID: GAM-003
需求名称: 每日情侣任务
优先级: P0
预估工时: 10 人日（后端 6，前端 3，算法 1）
```

#### 任务类型
每天早上 8:00 刷新 3 个当日任务。

**任务池示例**：

| 任务类型 | 示例 | 奖励 | 完成条件 |
|----------|------|------|----------|
| 互动型 | 今天给 TA 发送一次投喂 | 养分 +30 | feed_send |
| 记录型 | 一起新增一家想去的餐厅 | 养分 +40 | menu_add |
| 探索型 | 尝试一家从未去过的菜系 | 养分 +50 | menu_visit_new_cuisine |
| 回忆型 | 翻看一个月前的笔记并评论 | 养分 +30 | note_comment_old |
| 惊喜型 | 给对方写一条匿名树洞留言 | 养分 +40 | tree_hole_message |
| 同步型 | 双方同时在线 5 分钟 | 养分 +20 | online_together_5min |

#### 页面原型

```
任务卡片（首页中部）
┌─────────────────────────────────────┐
│ 今日任务                [0/3 完成]  │
│ ─────────────────────────────────── │
│ ☑ 给 TA 发送一次投喂                │
│    已完成 ✓  养分 +30               │
│ ─────────────────────────────────── │
│ ☐ 新增一家想去的餐厅                │
│    进度 0/1  养分 +40   [去完成]    │
│ ─────────────────────────────────── │
│ ☐ 双方同时在线 5 分钟               │
│    进度 2:31 / 5:00  养分 +20       │
│                                     │
│ [领取今日宝箱]（需完成 2 个解锁）   │
└─────────────────────────────────────┘
```

**任务生成算法**：
1. 从任务池中随机抽取 1 个「互动型」任务
2. 根据用户近 7 天行为缺口，抽取 1 个「补足型」任务（如：近 7 天未发笔记 → 抽到发笔记任务）
3. 随机抽取 1 个其他类型任务
4. 确保 3 个任务不重复，且难度适中

**技术方案**：
- 任务生成服务：每日 8:00 定时任务为所有活跃 couple 生成当日任务
- 任务状态存储：MySQL `daily_task` 表 + Redis 缓存今日任务
- 进度追踪：实时事件更新任务进度（如：发投喂时同时检查并更新任务状态）
- 宝箱奖励：伪随机算法，根据用户等级和付费状态调整掉落概率

#### 数据埋点方案

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| daily_task_expose | 任务卡片曝光 | task_type, task_position | 任务曝光 |
| daily_task_click | 点击任务 | task_type, task_id | 任务点击 |
| daily_task_progress_update | 任务进度更新 | task_type, current_progress | 任务进度 |
| daily_task_complete | 完成任务 | task_type, reward_amount | 任务完成率 |
| daily_task_box_open | 开启宝箱 | box_type, reward_item | 宝箱开启率 |
| daily_task_refresh | 刷新任务（会员） | refresh_count | 刷新使用率 |

### 3.3 情侣默契大考验

```
需求ID: GAM-005
需求名称: 情侣默契问答
优先级: P1
预估工时: 10 人日（后端 5，前端 4，题目库 1）
```

#### 玩法规则
1. 系统每天推送 3 道关于彼此的问题
2. 双方各自作答，答案同步揭晓
3. 答案一致获得「默契分」，不一致时展示正确答案
4. 每周统计默契率，默契率前 10% 的情侣获得「心有灵犀」限时勋章

**页面原型**：

```
默契问答页
┌─────────────────────────────────────┐
│ 今日默契问答  1/3                   │
│                                     │
│ Q: TA 最爱喝的奶茶口味是？          │
│                                     │
│ A. 珍珠奶茶                         │
│ B. 芋泥波波                         │
│ C. 杨枝甘露    ← 你的选择           │
│ D. 焦糖玛奇朵                       │
│                                     │
│ [提交答案]                          │
│         ↓（双方均提交后）           │
│ 揭晓页面：                          │
│ 🎉 答对啦！                         │
│ 你的答案：C. 杨枝甘露               │
│ TA 的答案：C. 杨枝甘露              │
│ 默契值 +10                          │
│ [下一题]                            │
└─────────────────────────────────────┘
```

**题目数据结构**：

```json
{
  "question_id": "q_001",
  "question_text": "TA 最爱喝的奶茶口味是？",
  "options": [
    {"option_id": "A", "text": "珍珠奶茶"},
    {"option_id": "B", "text": "芋泥波波"},
    {"option_id": "C", "text": "杨枝甘露"},
    {"option_id": "D", "text": "焦糖玛奇朵"}
  ],
  "correct_source": "user_preference",
  "source_table": "user_profile",
  "source_field": "favorite_drink"
}
```

**技术方案**：
- 题目池：MySQL `quiz_question` 表，分类管理（笔记提取、菜单数据、投喂记录、心愿单、通用题）
- 动态题目生成：对于「数据驱动题」，每日定时任务根据 couple 的最新数据生成当日专属题目
- 答题状态：Redis Hash 存储 `couple_quiz:{couple_id}:{date}`，记录双方答题进度
- 默契率计算：`sum(答对题数) / sum(总答题数) × 100%`

#### 数据埋点方案

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| quiz_entry | 进入问答页面 | quiz_date | 问答入口 UV |
| quiz_answer_submit | 提交答案 | question_id, selected_option | 答题提交率 |
| quiz_answer_reveal | 揭晓答案 | question_id, is_correct | 正确率 |
| quiz_weekly_report_view | 查看周报 | correct_rate_pct | 周报查看率 |
| quiz_share_poster | 分享默契证书 | correct_rate_pct, share_channel | 证书分享率 |

#### 勋章展示页面原型

```
勋章墙页面
┌─────────────────────────────────────┐
│ 已获得勋章  12 / 48                 │
│                                     │
│ [展示墙预览]                        │

│ 🥈 爱心大厨（100次）       ✓ 已获得 │

└─────────────────────────────────────┘
```

**勋章检测机制**：
- 实时检测：用户完成特定行为时（如：发送第 100 次投喂），即时发放勋章
- 定时检测：每日凌晨扫描所有 couple，检测基于时间累积条件的勋章（如：连续 7 天秒回）
- 发放通知：勋章获得时发送 Push + App 内消息 + 全屏弹窗动画

**技术方案**：
- 勋章定义：配置在 `medal_definition` 表，包含条件表达式（JSON）
- 条件引擎：Java 反射 + 简单脚本引擎执行条件判断
- 用户勋章：`user_medal` 表记录获得时间、展示顺序
- 勋章动画：Lottie 或视频片段，获得时播放 2-3 秒

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| medal_wall_view | 查看勋章墙 | medal_category | 勋章墙 UV |

优先级: P1

#### 报告页面原型

```
年度报告 H5（翻页式）
┌─────────────────────────────────────┐
│  ←  我们的 2025 年度报告  →         │
│                                     │
│ Page 1: 封面                         │

│                                     │
│ Page 4: 互动数据                     │
│ 投喂 87 次 | 消息 3,240 条           │
│ 相守 365 天 | 最长连续打卡 42 天     │
│                                     │
│ [生成海报分享] [保存图片]            │
└─────────────────────────────────────┘
```

**技术方案**：
- 数据聚合：12 月中下旬启动年度数据预计算任务，为所有 couple 生成报告草稿
- 报告生成：前端基于 JSON 数据动态渲染 H5 翻页组件（如：Swiper.js）
- 海报生成：服务端使用 Puppeteer/Playwright 截图生成报告海报（高保真）
- 高峰期容灾：报告生成采用异步队列，错峰处理；登录时若报告未生成，显示"正在飞奔而来"
