## 一、文档概述

### 1.1 背景
「AI Couple Dish」目前聚焦于「餐厅」这一单一场景。但情侣的生活远不止吃饭：一起旅行、一起制作家常菜、一起采购食材、一起规划周末……如果产品能从「约会餐厅助手」延伸为「情侣生活方式平台」，用户的使用场景将从「约会前」扩展到「日常中」，使用频次和用户生命周期价值（LTV）将大幅提升。

### 1.2 核心洞察
情侣的核心共同行为依次为：
1. **吃饭**（最频繁）→ 已有
2. **做饭/买菜**（每周 2-3 次）→ 待拓展
3. **短途旅行/周边游**（每月 1-2 次）→ 待拓展
4. **节日庆祝/互赠礼物**（每季度）→ 待拓展
5. **家居生活/共同兴趣**（长期）→ 待拓展

### 1.3 延伸策略
采用「同心圆」模型向外扩展：
```
核心圆：餐厅/美食（已有）
    ↓
第二圆：家常菜/食材采购/菜谱
    ↓
第三圆：旅行/打卡/足迹
    ↓
第四圆：礼物/节日/生活美学
```

**菜谱导入流程**：
1. 用户粘贴链接 → 前端校验 URL域名
2. 后端调用爬虫/解析服务（或第三方 API 如：万物皆数的菜谱解析服务）
3. 解析失败时，提供「手动输入」入口
4. 解析成功，用户可编辑后保存

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★☆ | 爬虫解析技术成熟，但需对抗反爬 |
| 实现复杂度 | 中等 | 需适配多个平台的页面结构变化 |
| 外部依赖 | 较多 | 下厨房/小红书/抖音解析 |
| 数据安全 | 低 | 公开菜谱数据 |
| 性能风险 | 低 | 异步解析，不影响主流程 |

**技术方案**：
- **菜谱解析**：
  - 首选方案：接入第三方内容解析 API（如：Jina AI Reader API 可解析任意网页为 Markdown，再提取结构化数据）
  - 备选方案：自建 Python 爬虫服务（Scrapy + Playwright），针对下厨房做定向解析
- **数据模型**：
  - `recipe` 表：菜谱基础信息
  - `recipe_ingredient` 表：食材（支持名称、用量、分类）
  - `recipe_step` 表：步骤（支持文字、图片、语音）
  - `recipe_cook_record` 表：做菜记录（时间、评分、照片）
- **语音播报**：使用微信小程序 `BackgroundAudioManager` 或 H5 Web Speech API
- **关联采购清单**：点击「今天做这个」时，读取 `recipe_ingredient` 生成待购清单项

**实时同步机制**：
- 使用 WebSocket 实现双方清单的实时同步
- 一方勾选/添加/删除时，另一方 200ms 内同步更新
- 离线时同步：用户上线后拉取最新清单快照

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★★ | CRUD + WebSocket 非常成熟 |
| 实现复杂度 | 中等 | 需设计食材标准化库、实时同步冲突处理 |
| 外部依赖 | 少 | 无强外部依赖 |
| 数据安全 | 低 | 采购清单无敏感信息 |
| 性能风险 | 低 | WebSocket 连接数可控 |

**技术方案**：
- **清单数据**：MySQL `shopping_list` 表 + `shopping_item` 明细表
- **实时同步**：Spring Boot + Netty WebSocket Server，消息通过 Redis Pub/Sub 多实例广播
- **食材标准化**：建立 `ingredient_dictionary` 基础食材库（约 500 种常见食材），支持别名映射
  - 如："西红柿"、「番茄」、「洋柿子」映射到同一个 ingredient_id
- **库存管理**：`pantry_inventory` 表记录家中库存，支持保质期字段 `expiry_date`
- **过期提醒**：每日定时任务扫描即将过期（3 天内）食材，推送提醒

  - 小程序：原生 `map` 组件
- **路线规划**：调用高德地图路径规划 API（步行/驾车/公交），根据行程项顺序连成路线
- **预算管理**：每个 schedule_item 支持绑定消费记录，自动汇总到 plan 级别
