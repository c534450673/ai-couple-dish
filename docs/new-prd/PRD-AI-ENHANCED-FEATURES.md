## 一、文档概述

### 1.1 背景
当前情侣应用市场同质化严重，功能多为信息记录工具。AI Couple Dish 项目的核心差异点不应仅停留在"记录"，而应升级为"智能侣伴"——通过 AI 技术预测需求、生成创意、主动提醒，让产品从被动工具变成情侣关系的主动促进者。

### 1.2 愿景
> 做最懂情侣的 AI 助手，让每一次互动都有惊喜，让每一段回忆都被温柔对待。

### 1.3 设计原则
1. **智能不打扰**：AI 在恰当时机出现，不制造信息噪音
2. **惊喜可预期**：让用户对 AI 的介入产生正向期待
3. **双向增益**：AI 的建议同时服务两个人，避免单方面输出
4. **隐私优先**：AI 训练数据本地化处理，核心情感数据不上云

## 二、AI 智能餐厅推荐引擎

### 2.1 需求背景
情侣约会最常见的对话："今天吃什么？""随便。""那吃火锅？""不想吃。"这种"选择困难症"每天在各处情侣间上演，是强痛点。

### 2.2 功能设计

#### 2.2.1 AI 猜你想吃

```
需求ID: AI-MEN-001
需求名称: AI 猜你想吃
优先级: P0
预估工时: 10 人日（后端 6，前端 2，测试 2）
```

**核心逻辑**：
- 基于历史菜单数据（收藏、点赞、去过次数、评分）
- 结合天气、时间、地理位置、季节
- 分析双方近期口味偏好变化
- 输出 3 个餐厅推荐 + 推荐理由

**推荐维度**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 历史偏好 | 35% |  TA 最爱的菜系、常去的区域 |
| 时间节点 | 20% |  纪念日 nearby → 推荐有仪式感的餐厅 |
| 天气季节 | 15% |  下雨天 → 火锅/居酒屋；夏天 → 日料/轻食 |
| 新鲜度 | 15% |  避免连续推荐同类型，引入 20% 新餐厅 |
| 实时位置 | 15% |  工作日午餐推荐公司附近，周末推荐商圈 |

**页面原型与交互流程**：

```
首页顶部 AI 推荐卡片
┌─────────────────────────────────────┐
│ 🤖 AI 猜你想吃                      │
│ 今天降温了，来顿热乎乎的火锅？      │
│ 【鸳鸯火锅】小龙坎 · 距你 1.2km     │
│ 查看今日推荐 →                      │
└─────────────────────────────────────┘
         ↓ 点击
推荐详情页（全屏底部弹出）
┌─────────────────────────────────────┐
│ 关闭                         🍲     │
│ 基于今天周五+降温5°C，AI 推荐：     │
│ ─────────────────────────────────── │
│ [推荐1] 小龙坎火锅                  │
│ "你们上次给这里打了5星，今天降温    │
│  正好适合来顿热乎的~"               │
│ [预约] [加入心愿单] [不感兴趣]      │
│ ─────────────────────────────────── │
│ [推荐2] 凑凑火锅                    │
│ ...                                 │
│ ─────────────────────────────────── │
│ [推荐3] 居酒屋·鸟贵族               │
│ ...                                 │
│                                     │
│ [就这家] 发送给 TA                  │
└─────────────────────────────────────┘
```

**交互流程**：
1. 用户打开首页，AI 推荐卡片基于算法预计算结果展示在顶部第 2 个卡片位
2. 点击后展开推荐详情页，展示 3 个推荐项
3. 用户可点击「不感兴趣」反馈，系统记录并 7 天内不再推荐同类
4. 点击「就这家」后推送通知给对方："TA 看中了【XX 餐厅】，你觉得怎么样？"

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★☆ | 推荐系统技术成熟，规则引擎+协同过滤即可 MVP |
| 实现复杂度 | 中等 | 需建立用户-餐厅评分矩阵、特征工程、实时位置获取 |
| 外部依赖 | 中等 | 天气 API（高德/彩云）、地理位置服务、大模型 API |
| 数据安全 | 中等 | 餐厅偏好数据脱敏后可云端处理，无高风险 |
| 性能风险 | 低 | 推荐结果可每日凌晨预计算并缓存，无需实时大计算 |

**技术方案**：
- **推荐服务**：新建 `ai-recommendation-service`，与主业务解耦；MVP 阶段使用 Java + Redis 实现规则引擎
- **特征工程**：基于 MySQL 视图或定时任务生成用户-餐厅特征表
- **推荐理由生成**：调用 Claude API / 通义千问，输入模板化 prompt，输出文案后本地缓存 24h
- **缓存策略**：每对情侣的推荐结果按 couple_id 缓存到 Redis，TTL=4h；地理位置变化时失效重算
- **降级策略**：AI API 不可用时，回退到基于规则的人工文案库（约 100 条模板）

| ai_recommend_click | 用户点击推荐卡片 | couple_id, recommend_scene, click_position | 推荐点击率 |
| ai_recommend_detail_view | 进入推荐详情页 | couple_id, recommend_id, weather_tag | 详情页 UV |

| ai_recommend_wishlist_click | 点击「加入心愿单」 | couple_id, restaurant_id | 推荐转化-心愿 |
| ai_recommend_send_partner | 点击「就这家/发送TA」 | couple_id, restaurant_id, partner_response | 推荐转化-分享 |

#### 2.2.2 拍照即识餐厅

```
需求ID: AI-MEN-002
需求名称: 拍照即识餐厅
优先级: P1
预估工时: 8 人日（后端 4，前端 2，AI 对接 2）
```

**场景**：用户上传约会照片或菜单照片，系统自动识别餐厅和菜品信息。

**功能点**：
1. **餐厅识别**：通过门头照/环境照识别餐厅名称（OCR + 图像识别）
2. **菜品识别**：识别菜单照片上的菜品和价格
3. **信息自动填充**：识别到餐厅名后，自动搜索大众点评/高德 POI 数据补充地址、电话、人均
4. **发票识别**：上传餐饮发票自动提取餐厅名、金额、日期，生成消费记录

**页面原型与流程**：

```
添加菜单页
┌─────────────────────────────────────┐
│ 新增餐厅                            │
│ ┌─────────┐ ┌─────────┐            │
│ │ 相册    │ │ 📷 拍照 │            │
│ └─────────┘ └─────────┘            │
│                                     │
│ [上传照片后显示识别中动画]            │
│  ⚡ AI 正在识别中...                 │
│                                     │
│ 识别结果：                           │
│ 餐厅名称：海底捞火锅 [✏️可修改]       │
│ 地址：上海市静安区XX路 [✏️可修改]     │
│ 人均：¥128                           │
│ 菜品：毛肚 ¥48、鸭肠 ¥32 [+添加]     │
│                                     │
│        [确认并保存]                 │
└─────────────────────────────────────┘
```

**交互流程**：
1. 用户在添加菜单页点击「拍照/选图」
2. 图片上传至 OSS 后触发异步识别任务
3. 前端轮询识别结果（3-5 秒超时回退到手动输入）
4. 识别结果回填到表单，用户确认后保存

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★★ | 阿里云/百度 OCR 已非常成熟，通用发票识别准确率 > 95% |
| 实现复杂度 | 中等 | 需要 POI 数据匹配、异步任务调度、前端轮询优化 |
| 外部依赖 | 较多 | OCR API、POI 搜索 API（高德/大众点评） |
| 数据安全 | 低 | 照片用于识别即删或按需保留，正常业务范畴 |
| 性能风险 | 低 | 异步处理，不影响主流程 |

**技术方案**：
- **OCR 服务**：阿里云通用文字识别 + 增值税发票识别
- **图像分类**：阿里云图像标签服务识别是否为"餐厅门头/菜单/发票"
- **POI 匹配**：调用高德地图关键词搜索 API，取最相似结果（基于编辑距离+地理位置）
- **异步任务**：Spring Boot + @Async 或 RabbitMQ 处理识别任务
- **存储**：原图存 OSS，识别结果存 MySQL；用户删除记录时同步删除原图

**数据埋点方案**：

|--------|----------|----------|----------|
| ai_ocr_upload | 用户上传图片 | image_type, source_page | 上传量 |
| ai_ocr_success | 识别成功返回结果 | image_type, ocr_provider, duration_ms | 识别成功率 |
| ai_ocr_fail | 识别失败或超时 | image_type, fail_reason | 失败率 |
| ai_ocr_poi_match | POI 匹配成功 | match_score, poi_source | POI 匹配率 |
| ai_ocr_form_edit | 用户修改识别结果 | edit_field_count | 识别准确率 |
| ai_ocr_save | 用户确认保存 | restaurant_id, save_time_s | 识别-保存转化率 |

## 三、AI 情感分析师

### 3.1 需求背景
产品有笔记模块和消息中心，但记录的内容是"死的"。如果能通过 AI 分析情感趋势，及时预警关系低谷，将产生巨大的用户价值。

### 3.2 功能设计

#### 3.2.1 恋爱情绪温度计

```
需求ID: AI-EMO-001
需求名称: 恋爱情绪温度计
优先级: P1
预估工时: 10 人日（后端 6，数据分析 2，前端 2）
```

**核心逻辑**：
- 分析双方近 30 天的互动数据（投喂频次、消息回复速度、笔记提及对方次数、纪念日互动）
- 输出一个可视化的「恋爱温度指数」（0-100 分）
- 每周一生成周报，包含趋势曲线和温馨建议

**页面原型**：

```
首页温度卡片
┌─────────────────────────────────────┐
│ 本周恋爱温度：78°  [趋势图小图标]   │
│ 🔥 比上周高了 3°，继续保持！        │
└─────────────────────────────────────┘
         ↓
温度详情页
┌─────────────────────────────────────┐
│ 本周恋爱温度                         │
│                                     │
│    78°                               │
│  [温度计大图标，动态填充]             │
│                                     │
│ 近4周趋势：                          │
│  72° → 75° → 78° [折线图]          │
│                                     │
│ 本周 Highlights：                    │
│ ✨ 互相投喂了 5 次，超过 82% 的情侣  │
│ ✨ 连续 7 天都有互动                 │
│ ✨ TA 在 2 篇笔记里提到了你          │
│                                     │
│ AI 建议：                            │
│ "最近两个人都很忙，推荐周末安排      │
│  一次有仪式感的约会放松一下~"        │
│                                     │
│ [查看详细分析] [关闭情绪分析]        │
└─────────────────────────────────────┘
```

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★☆ | 数据指标计算成熟，情感分析可用轻量 NLP |
| 实现复杂度 | 中等 | 需聚合多表数据、设计合理的评分算法 |
| 外部依赖 | 少 | 主要依赖内部数据，NLP 可用开源模型 |
| 数据安全 | 高 | 情感数据敏感，需严格限制访问范围 |
| 性能风险 | 低 | 可定时任务离线计算，无需实时大计算 |

**技术方案**：
- **数据聚合**：定时任务（每日凌晨 2:00）计算前一日各维度指标，写入 `couple_emotion_score` 表
- **评分算法**：采用加权线性模型，公式：`Score = Σ(指标值 × 权重)`，结果归一化到 0-100
- **NLP 情感分析**：
  - 笔记内容情绪分析使用本地化的 HanLP / Jieba + 自定义情感词典
  - 或者调用阿里云 NLP 情感分析 API（文本脱敏后）
- **周报生成**：周一早上 8:00 触发周报推送；AI 建议由模板 + 轻量规则生成，复杂场景才调用 LLM
- **隐私保护**：情绪分析数据 couple_id 级别隔离，不对外暴露任何排名

**数据埋点方案**：

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| emotion_score_card_expose | 温度卡片出现在首页 | current_score, week_over_week | 卡片曝光 |
| emotion_score_card_click | 点击温度卡片 | current_score | 卡片点击 |
| emotion_score_detail_view | 查看详情页 | current_score, trend_direction | 详情页 UV |
| emotion_score_close_click | 点击关闭情绪分析 | close_reason | 关闭率 |
| emotion_weekly_report_view | 查看周报 | report_week | 周报查看率 |
| emotion_weekly_report_share | 分享周报 | share_channel | 周报分享率 |

#### 3.2.3 吵架和解助手

```
需求ID: AI-EMO-003

优先级: P2
预估工时: 8 人日（后端 4，前端 2，AI prompt 2）
```

**场景**：情侣吵架后冷战，双方都不好意思先开口。AI 作为中立第三方，提供破冰方案。

**触发方式**：
- 用户主动点击「求和好」按钮
- 系统检测到双方互动数据连续 3 天异常低迷时，推送温和提醒

```

│                                     │
│ [破冰文案]                           │
│ "我打开 APP 看到我们上周去           │
│  【XX店】的照片，突然很想再吃一次。  │
│  这次我保证不抢你碗里的肉。"         │
│ [复制发给 TA] [换一条]              │
│                                     │
│ [推荐和解餐厅]                       │

│                                     │
│ [回忆杀] 看看这些美好瞬间 →          │
└─────────────────────────────────────┘
```

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★☆☆ | 检测算法成熟，但破冰文案需精心设计避免越界 |
| 实现复杂度 | 中等 | 需准确定义"低迷"阈值，文案需大量测试 |
| 外部依赖 | 中等 | LLM API |
| 数据安全 | 高 | 关系状态属于高度隐私，推送时机和内容需极其谨慎 |
| 性能风险 | 低 | 离线检测 + 低频触发 |

**技术方案**：
- **低迷检测规则**：连续 N 天无消息+无投喂+无笔记互动
- **干预策略**：仅向"最近有登录行为"的一方推送，避免打扰完全离线的一方
- **文案生成**：使用温和、略带幽默的 prompt，避免说教感；设置内容审核过滤层
- **餐厅推荐**：从双方共同高评分餐厅中随机选 1 家

| 评估项 | 评分/等级 | 说明 |

## 六、AI 隐私与安全设计

### 6.1 数据分级处理

| 数据级别 | 处理方式 | 示例 |
|----------|----------|------|
| 公开级 | 可上传云端 | 餐厅 POI 信息、天气数据 |
| 敏感级 | 脱敏后云端处理 | 用户行为特征、互动频次 |
| 私密级 | 仅本地处理 | 笔记原始文本、语音内容、照片 |

### 6.2 AI 功能开关
- 每个 AI 功能均可独立开启/关闭
- 关闭后不影响基础功能使用
- 首次使用 AI 功能前需二次确认隐私授权

### 6.3 生成内容审核
- 所有 AI 生成的对外分享内容经过敏感词过滤
- 用户可举报不当生成内容
