## 一、文档概述

### 1.1 背景
情侣类应用天然具备社交传播属性：甜蜜瞬间值得炫耀、有趣互动乐于分享、关系里程碑具有仪式感。然而，当前产品缺少系统化的裂变设计和病毒式传播机制，用户增长主要依赖自然流量，获客成本未被有效分摊。

### 1.2 增长目标

| 指标 | 当前（预估） | 6 个月目标 | 12 个月目标 |
|------|-------------|-----------|------------|
| 新增用户占比（老带新） | 5% | 35% | 50% |
| 分享率（月活用户产生分享行为占比） | 2% | 20% | 30% |
| 邀请转化率（点击邀请链接后注册绑定率） | - | 15% | 20% |
| viral K-因子 | < 0.1 | > 0.3 | > 0.5 |

### 1.3 增长策略
采用 **AARRR + PLG（Product-Led Growth）** 双轮驱动。

### 2.2 动态视频生成

```
需求ID: VRL-CNT-002
需求名称: 恋爱回忆短视频生成
优先级: P1
预估工时: 14 人日（后端 8，前端 4，视频生成 2）
```

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★☆☆ | 服务端视频剪辑可用 FFmpeg，但模板化混剪需较多调试 |
| 实现复杂度 | 较高 | 需要音视频同步、转场特效、字幕渲染 |
| 外部依赖 | 中等 | 可选接入剪映/即梦 API 降低自研成本 |
| 数据安全 | 低 | 无敏感信息 |
| 性能风险 | 高 | 视频生成耗时 10-60s，高峰期需排队处理 |

**技术方案**：
- **MVP 方案**：接入剪映开放平台的「视频合成 API」，传入图片、文案、BGM ID，直接获取成品视频
- **自研方案**（远期）：FFmpeg + Python MoviePy 模板化剪辑
  - 模板定义：镜头时长、转场类型、字幕位置、BGM 轨道
  - 图片处理：统一裁切为 9:16
  - 字幕生成：将纪念日文案按时间轴插入
- **异步队列**：视频生成任务写入 RabbitMQ，生成完成后 Push 通知用户
- **BGM 库**：采购 20-50 首无版权音乐，按风格分类（甜蜜/治愈/港风/电子）

#### 数据埋点方案

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| video_generate_start | 开始生成 | video_type, photo_count | 视频生成量 |
| video_generate_success | 生成成功 | video_type, duration_sec | 生成成功率 |
| video_play | 播放视频 | video_type | 播放率 |
| video_share_douyin | 分享到抖音 | video_type | 抖音分享率 |
| video_share_wechat | 分享到微信 | video_type | 微信分享率 |

#### 邀请链路

```
老用户生成专属邀请码/链接
    ↓
分享到微信/朋友圈/抖音/小红书
    ↓
新用户点击 → 下载 App / 打开小程序
    ↓
新用户注册 + 绑定情侣
    ↓
双方获得奖励
```

#### 落地页设计

```
邀请落地页（H5 / 小程序）
┌─────────────────────────────────────┐
│                                     │
│    [老用户头像] [爱心] [TA头像]      │
│    "张三和李四已经在一起 365 天"     │
│    "快来测测你和 TA 的默契值吧！"    │
│                                     │
│    [输入手机号] [获取验证码]         │
│    [立即加入]                       │
│                                     │
│    已有 12,345 对情侣在使用          │
│                                     │
└─────────────────────────────────────┘
```

**技术可行性分析**：

| 评估项 | 评分/等级 | 说明 |
|--------|----------|------|
| 技术成熟度 | ★★★★★ | 邀请码体系非常成熟 |
| 实现复杂度 | 中等 | 需追踪全链路转化、防刷、奖励发放幂等 |
| 外部依赖 | 中等 | 微信小程序分享、短信服务 |
| 数据安全 | 中等 | 需防止 invite bot 刷量 |
| 性能风险 | 低 | 常规业务逻辑 |

**技术方案**：
- **邀请码生成**：6 位字母数字组合，关联到 `user_invite_code` 表
- **归因追踪**：
  - H5：URL 中携带 `?inviter_code=ABC123` 写入 Cookie/LocalStorage
  - 小程序：`scene` 参数写入全局状态
  - App：Universal Link / App Link 携带参数
- **转化判定**：新用户注册时检查是否有 invite_code，写入 `user_referral` 表
- **奖励发放**：绑定情侣后触发奖励发放，使用分布式锁保证幂等
- **防刷策略**：同一手机号/IP/设备 ID 短期内只能被邀请一次；邀请量异常时触发风控

```
需求ID: VRL-PSH-001
需求名称: 智能消息推送策略
优先级: P0
预估工时: 8 人日（后端 6，前端 1，策略 1）
```

**技术方案**：
- **推送通道**：
  - App：个推/极光推送
  - 小程序：微信订阅消息 + 服务号模板消息
  - H5：浏览器 Web Push（Chrome）+ 短信 fallback
- **推送平台**：自建 Push 中心，统一管理各通道消息模板、发送记录、回执统计
- **个性化文案**：使用用户昵称、对方昵称、最近数据动态填充
- **频次控制**：每用户每日最多 5 条 Push，同类 Push 最小间隔 4h
- **A/B 测试**：对同一类推送准备 2-3 条文案，随机分流测试点击率

**Push 发送状态流转**：
```
消息创建 → Push中心排队 → 通道发送 → 设备接收 → 用户点击 → 跳转目标页
    ↓           ↓           ↓          ↓           ↓
  写入DB    频次检查    回执记录    到达率     点击率
```

#### 数据埋点方案

| 事件名 | 触发时机 | 事件属性 | 业务指标 |
|--------|----------|----------|----------|
| push_send | 发送 Push | push_type, channel | 发送量 |
| push_arrive | Push 到达设备 | push_type, channel | 到达率 |
| push_click | 点击 Push | push_type, channel, ab_group | 点击率 |
| push_dismiss | 忽略/清除 Push | push_type | 忽略率 |

## 六、数据中台与增长实验

### 6.1 增长指标体系

| 层级 | 指标 | 计算方式 |
|------|------|----------|
| 曝光层 | 分享次数 / 邀请链接点击次数 | 每日统计 |
| 点击层 | 分享点击率 / 邀请页转化率 | 点击÷曝光 |
| 下载层 | 点击到下载转化率 | 下载÷点击 |
| 注册层 | 下载到注册转化率 | 注册÷下载 |
| 绑定层 | 注册到绑定情侣转化率 | 绑定÷注册 |
| 激活层 | 绑定后 7 天内完成首次投喂比例 | 首次投喂÷绑定 |
| 留存层 | 次日 / 7 日 / 30 日留存 | 标准留存 |
| 变现层 | 老带新用户的 ARPU / LTV | 收入÷新增 |

### 6.2 增长实验机制

每双周运行 1-2 个增长实验：

| 实验编号 | 实验假设 | 目标指标 | 技术实现 |
|----------|----------|----------|----------|
| EXP-001 | 将邀请奖励从「小星星」改为「现金红包」，邀请转化率提升 20% | 邀请转化率 | A/B 分流，对照组control vs 实验组cash |
| EXP-002 | 在首页增加「今日适合做分享的事」卡片，分享率提升 15% | 分享率 | 50% 用户暴露卡片，对比点击和分享 |
| EXP-003 | 分手后 30 天推送「新的恋爱故事等你开启」，召回率提升 10% | 7 日召回率 | 对解绑用户实验组发送 vs 对照组不发送 |
| EXP-004 | 首次绑定后进入「甜蜜任务」而非直接进入首页，7 日留存提升 20% | 7 日留存 | 新用户绑定后随机分配 onboarding 路径 |

**A/B 测试技术方案**：
- 实验平台：MVP 阶段使用自建 `experiment_assignment` 表
- 分流逻辑：基于 `user_id` hash 取模，确保同一用户多次进入实验组不变
- 实验周期：最少 7 天，样本量对照组和实验组各不少于 1000 人
- 数据回收：通过埋点事件中的 `experiment_id` 和 `group_id` 属性进行分组统计
